# рҹ§  Big Picture Difference

| Codebase            | Architecture Style                                       | Key Idea                                   |
| ------------------- | -------------------------------------------------------- | ------------------------------------------ |
| **First Codebase**  | **Traditional Layered / N-Tier Architecture**            | Code organized by **technical layers**     |
| **Second Codebase** | **Clean Architecture + Vertical Slice + CQRS (MediatR)** | Code organized by **features / use cases** |

Think of it like this:

* **First project:** "Where should this code go technically?"
* **Second project:** "Which *feature* does this code belong to?"

That one difference changes **everything**.

---

# рҹҸ— FIRST CODEBASE

### Classic Layered / N-Tier Architecture

Structure simplified:

```
API (Controllers)
     вҶ“
BusinessLayer (Services)
     вҶ“
RepositoryLayer
     вҶ“
ModelLayer
```

Plus:

```
Domain
Infrastructure
Tests
```

### Typical Layer Responsibilities

| Layer           | Responsibility            |
| --------------- | ------------------------- |
| API             | HTTP endpoints            |
| BusinessLayer   | Business logic            |
| RepositoryLayer | Data access               |
| ModelLayer      | DTOs / Entities           |
| Domain          | Domain rules              |
| Infrastructure  | Technical implementations |

---

### Example request flow

Suppose user calls:

```
POST /convert
```

Flow becomes:

```
Controller
   вҶ“
BusinessLayer Service
   вҶ“
Repository
   вҶ“
Model / Entity
```

This is the **classic textbook architecture**.

---

### Key characteristics

вң” Layer-based organization
вң” Services hold logic
вң” Repositories handle persistence
вң” DTOs move data
вң” Controllers call services

But it has common problems:

рҹҡЁ **God Services**

```

```

becomes massive over time.

рҹҡЁ **Feature logic scattered**

Example: Convert Quantity feature might exist in:

```
Controller
Service
Repository
DTO
Model
```

spread across **5 folders**.

Developers start playing:

> "Where the hell is the code for this feature?"

# рҹ”Һ Inside Application

```
Application
 в”ңв”Җв”Җ DTOs
 в””в”Җв”Җ Features
      в””в”Җв”Җ Quantities
            AddQuantitiesCommand
            AddQuantitiesHandler
            CompareQuantitiesQuery
            CompareQuantitiesHandler
```

This is **Vertical Slice Architecture**.

Each feature is implemented as:

```
Command/Query + Handler
```

This pattern comes from **CQRS**.

---

# вҡЎ CQRS Explained Simply

CQRS =

**Command Query Responsibility Segregation**

Meaning:

| Type    | Purpose      |
| ------- | ------------ |
| Command | Change state |
| Query   | Read data    |

Examples in your project:

Commands:

```
AddQuantitiesCommand
DivideByScalarCommand
SubtractQuantitiesCommand
```

Queries:

```
CompareQuantitiesQuery
ConvertQuantityQuery
DivideQuantitiesQuery
```

---

# рҹ”Ғ Handler Pattern

Each operation has a handler.

Example:

```
AddQuantitiesCommand
      вҶ“
AddQuantitiesHandler
```

The handler contains the logic.

This is usually triggered through **MediatR**.

You actually have this dependency:

```
MediatR.dll
MediatR.Extensions.Microsoft.DependencyInjection.dll
```

So flow becomes:

```
Controller
   вҶ“
MediatR.Send()
   вҶ“
Command/Query
   вҶ“
Handler
   вҶ“
Domain
```

---

# рҹ§© Example Flow (Second Project)

Request:

```
POST /quantities/add
```

Flow:

```
QuantitiesController
     вҶ“
Mediator.Send(AddQuantitiesCommand)
     вҶ“
AddQuantitiesHandler
     вҶ“
Domain Model
```

No giant service class.

Each feature is **isolated**.
